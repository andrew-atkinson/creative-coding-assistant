# Audio-Reactive Particle Systems with p5.js

This tutorial explores how to move beyond simple, direct spectrogram visualizations (like a standard graphic equalizer) toward more complex, generative art. We will achieve this by using Fast Fourier Transform (FFT) data to drive the lifecycle and behavior of custom objects.

## Moving Beyond Direct Visualization [00:00]

A standard spectrogram visualization often feels "trapped" by its own structure. In a basic implementation, you might see vertical light streaks or circles that simply change size based on frequency. While effective for showing raw data, these objects lack "life"—they are purely reactive and have no persistence.

The current implementation in the editor uses `fft.analyze()` to create a basic spectrogram:

```javascript
// Basic spectrogram implementation in draw()
bins = fft.analyze();
for (let y = 0; y < bins.length; y++) {
  fill(bins[y], 50); // Uses frequency amplitude for color/transparency
  rect(x, y, 1, 1);
}
```

The goal is to transition from these "symptoms" of sound into a system where audio acts as a seed for persistent, moving objects.

## Analyzing Audio with Octave Bands [01:00]

To create more musically intuitive visualizations, we shouldn't just look at raw frequency bins. Instead, we can use **Octave Band Analysis**. This groups frequencies into logarithmic segments (typically 11 octaves), which aligns better with how humans perceive musical pitch.

To implement this, we use `fft.getOctaveBands()` and then calculate the average amplitude for each of those octaves using `fft.logAverages()`.

```javascript
// Extracting octave-based data
octaveBands = fft.getOctaveBands(1); 
octaveAverages = fft.logAverages(octaveBands);

// Visualizing the octave averages as circles
push();
for (let i in octaveAverages) {
  // Map the index to a vertical position and color gradient
  fill(0, (i / octaveAverages.length) * 255, 15);
  circle(width / 2, (i / octaveAverages.length) * height, 1);
}
pop();
```

**Key Insight:** By using the index `i` divided by the total length of the array, we can map data to a normalized range (0.0 to 1.0). This allows us to translate mathematical data into spatial positions and color gradients across the canvas height.

## Implementing Object-Oriented Animation [02:36]

To create "persistent" noise—objects that exist on screen even as the sound fluctuates—we need to move from drawing simple shapes to managing a collection of objects via a Class.

### Creating the Node Class
We will create a new file, `node.js`, to define our `Node` class. Instead of random placement, we want these nodes to be spawned at specific coordinates and colors derived from the audio data.

```javascript
// node.js
class Node {
  constructor(c, x, y, amp) {
    this.x = x;
    this.y = y;
    // We will eventually use 'amp' to influence speed
    this.xSpeed = random(-1.5, 1.5);
    this.ySpeed = random(-1.5, 1.5);
    this.color = c;
  }

  move() {
    // Basic boundary bouncing logic
    if (this.x < 0 || this.x > width) {
      this.xSpeed *= -1;
    }
    if (this.y < 0 || this.y > height) {
      this.ySpeed *= -1;
    }
    this.x += this.xSpeed;
    this.y += this.ySpeed;
  }
}
```

### Linking Audio to Object Behavior
The power of this approach lies in the `constructor`. By passing audio-derived values into a new instance of a class, we can create direct correlations:
1.  **Position ($x, y$):** A node is spawned at the specific coordinate of a frequency peak.
2.  **Color:** The node inherits the color assigned to that specific octave band.
3.  **Velocity (Speed):** The amplitude of the sound can be used as a multiplier for `xSpeed` and `ySpeed`, meaning louder sounds create faster, more energetic particles.

### Integrating the Class
To use this class in your main sketch, you must ensure it is linked via `index.html` and then instantiated within the `draw()` loop based on audio triggers.

First, verify the class is accessible by logging it to the console:
```javascript
// In sketch.js
console.log(Node); // Should output: Node() {}
```

Once verified, you can begin the process of "spawning" these nodes when audio amplitude crosses a certain threshold, creating a burst effect that responds to the music.

# Audio-Reactive Particle Systems with Octave Bands

This tutorial explores how to move beyond simple spectrogram visualizations by using Fast Fourier Transform (FFT) analysis to drive complex, object-oriented animations. We will transition from drawing simple shapes based on frequency bins to managing a system of "nodes" (particles) that are created and destroyed in response to specific musical octaves.

## Analyzing Audio with Octave Bands [06:00]

While standard frequency bins provide a granular view of sound, they can be too "noisy" for artistic purposes. To create more musically intuitive visualizations, we use **octave bands**. This groups frequencies into logarithmic segments, allowing the visualization to respond more naturally to different musical instruments (e.g., distinguishing bass from a violin).

To implement this in p5.js, you must first perform the FFT analysis before you can extract octave data.

```javascript
// 1. Perform the initial analysis to get frequency bins
bins = fft.analyze();

// 2. Get the octave bands based on that analysis
octaveBands = fft.getOctaveBands(1);

// 3. Get the logarithmic averages for those bands
octaveAverages = fft.getAverages(octaveBands);

// Log the results to the console to verify data
console.log(octaveAverages);
```

### Visualizing the Octave Data [06:35]

Once we have `octaveAverages`, we can map these values to visual properties. In the example below, we use a loop to draw circles in the center of the screen. The size of each circle is driven by its corresponding octave average, and the color shifts across the spectrum based on its index.

```javascript
push();
for (let i in octaveAverages) {
  // Map the index to a color gradient and position
  fill(50, 0, (i / octaveAverages.length) * 255, 15);
  circle(width / 2, (i / octaveAverages.length) * height, octaveAverages[i]);
}
pop();
```

The console output for `octaveAverages` typically shows an array of numbers (e.g., `[95, 96, 73, 105...]`). These numbers represent the amplitude of each octave.

## Implementing Object-Oriented Animation [08:00]

To create a "spraying" or "bursting" effect, we move away from drawing simple shapes and instead use a `Node` class. This allows each visual element to have its own properties, such as position, velocity, and color.

### The Node Class Structure [08:30]

Each `Node` is initialized with a random position and velocity. This creates organic, non-linear movement.

```javascript
class Node {
  constructor(c) {
    this.x = random(width);
    this.y = random(height);
    this.xSpeed = random(-1.5, 1.5);
    this.ySpeed = random(-1.5, 1.5);
    this.color = c;
  }

  // Methods for movement and drawing would be defined here
  move() { /* ... */ }
  drawShapes(nodes) { /* ... */ }
}
```

### Managing the Lifecycle of Nodes [08:10]

Instead of creating a static number of objects in `setup()`, we manage their lifecycle within the `draw()` loop based on audio amplitude. This creates a dynamic system where particles are "born" when the music is loud and "die" when it becomes quiet.

*   **Creation:** If `octaveAverages[i]` exceeds a certain threshold, we `push()` a new `Node` into our array.
*   **Destruction:** If the amplitude drops below a threshold, we `shift()` (remove) nodes from the array.

```javascript
// Logic inside draw() loop
for (let i in octaveAverages) {
  if (octaveAverages[i] > 100) {
    // Add a node when the sound is loud enough
    nodes.push(new Node(color(r, g, b)));
  } else if (octaveAverages[i] < 150) {
    // Remove the oldest node when it gets too quiet to prevent clutter
    nodes.shift(); 
  }
}
```

## Using Two-Dimensional Arrays for Frequency Mapping [09:10]

To ensure that specific musical frequencies control specific groups of particles, we use a **two-dimensional (nested) array**. 

A standard one-dimensional array is just a list of values. A two-dimensional array is an "array of arrays." This allows us to map one index in our main `nodes` array to an entire collection of particles belonging to a specific octave band.

### Understanding Nested Arrays [09:30]

In a nested array, the first index accesses the sub-array, and the second index accesses the specific value within that sub-array.

```javascript
// A 2D array where each inner array represents a group of nodes
let nodes = [
  [1, 2, 3], // Index 0: Bass octave nodes
  [5, 6, 7], // Index 1: Low-mid octave nodes
  [3, 5, 7], // Index 2: Mid octave nodes
  [0, 2, 3]  // Index 3: High octave nodes
];

// Accessing data:
// nodes[0] refers to the entire first array: [1, 2, 3]
// nodes[0][2] refers to the third value in that first array: 3
```

### Mapping Octaves to Node Arrays [11:20]

By structuring our data this way, we can iterate through the `octaveAverages` and interact with a corresponding sub-array of nodes for each octave. This creates a direct, organized link between the audio data and the generative particle system.

```javascript
// Example structure for mapping octaves to node groups
let nodes = [
  [], // Array for bass octave bands
  [], // Array for low-mid octave bands
  [], // Array for mid octave bands
  // ... and so on
];
```

## Analyzing Octave Bands [12:02]

To create more musically intuitive visualizations, we move away from linear frequency bins and instead use logarithmic octave bands. This allows us to group frequencies into segments that correspond better to how humans perceive pitch.

Using the `p5.FFT` object, we can extract these bands and calculate their average amplitude:

```javascript
// Get octave bands (the number of bands can be adjusted)
octaveBands = fft.getOctaveBands(1);

// Calculate the average amplitude for each band
octaveAverages = fft.logAverages(octaveBands);

// Debugging: Check the values in the console
console.log(octaveAverages);
```

The `octaveAverages` array contains the amplitude values for each band. We can use this data to drive visual elements, such as circles whose size is determined by the amplitude and whose vertical position is mapped to their index in the array.

## Managing Complex Data with Nested Arrays [13:00]

When moving from simple shapes to a system of independent objects (nodes), we need a way to track which audio frequency controls which group of objects. This requires a **two-dimensional array** (an array of arrays).

### Initializing the Data Structure [13:02]
Because we are looping through `octaveAverages` to create objects, we must ensure that each index in our main `nodes` array is initialized as an empty array before we attempt to add objects to it. Attempting to `push()` an object into a non-existent index will cause errors.

```javascript
// nodes is our main array of arrays
let nodes = []; 

for (let i in octaveAverages) {
  // Check if the sub-array for this specific frequency band exists
  if (nodes[i] === undefined) {
    // If it doesn't exist, initialize it as an empty array
    nodes[i] = [];
  }
}
```

### Triggering Object Creation [14:30]
To create a "burst" or "spray" effect, we only want to spawn new objects when the audio reaches a certain intensity. We also need to implement "garbage collection" to prevent performance degradation; if we spawn objects 60 times a second without removing them, the array will grow indefinitely and slow down the application.

The following logic checks if a frequency peak has occurred and ensures the number of active objects in that band stays within a reasonable limit:

```javascript
for (let i in octaveAverages) {
  // 1. Check if the amplitude is high enough to trigger a new node (e.g., > 150)
  // 2. Check if the current sub-array is getting too long (e.g., > 20)
  if (octaveAverages[i] > 150 && nodes[i].length < 20) {
    
    // Calculate a color based on the frequency index (i / length)
    // This creates a gradient from blue to red across the spectrum
    let r = 50;
    let g = 0;
    let b = (i / octaveAverages.length) * 255;
    let colorValue = color(r, g, b);

    // Push a new instance of our Node class into the sub-array
    nodes[i].push(new Node(colorValue));
  }
}
```

## Mapping Data to Visual Properties [16:30]

A key part of generative art is mapping mathematical values to aesthetic properties. In the code above, we use a normalized fraction—the current index divided by the total length of the array—to create a gradient.

```javascript
// The fraction (i / octaveAverages.length) results in a value between 0 and 1
// This is used to map the index to a color spectrum or spatial position
let fraction = i / octaveAverages.length;

// Example: Mapping to color (Blue to Red)
fill(50, 0, fraction * 255, 15);

// Example: Mapping to vertical position
let yPos = fraction * height;
circle(width / 2, yPos, octaveAverages[i]);
```

By using this normalized value, the visualization remains consistent regardless of how many octave bands are being analyzed. When `i` is 0, the fraction is 0; when `i` reaches the end of the array, the fraction approaches 1.

# Managing Object Lifecycles in Audio-Reactive Systems

This lesson covers the implementation of a data-driven particle system where audio frequency data dictates when visual objects are created and destroyed. We will move from simple spectrograms to a complex system using nested arrays and Object-Oriented Programming (OOP).

## Managing the Lifecycle of Nodes [17:00]

To create a "burst" or "spray" effect in response to music, we cannot simply draw shapes; we must manage the lifecycle of discrete objects. This requires two main actions: adding new objects when audio amplitude is high and removing old objects to prevent memory leaks and visual clutter.

### Initializing Nested Arrays
Because we want different audio frequencies to control different sets of particles, we use a **two-dimensional array** (an array of arrays). Each index in the outer `nodes` array corresponds to a specific octave band.

```javascript
// Inside the draw loop
for (let i in octaveAverages) {
  // If this frequency band doesn't have an array yet, initialize it
  if (nodes[i] === undefined) {
    nodes[i] = [];
  }

  // If the audio is loud enough and we haven't reached a limit, add a new node
  if (octaveAverages[i] > 150 && nodes[i].length < 20) {
    // We pass a color object to the Node constructor
    nodes[i].push(color((i / octaveAverages.length) * 255, 0, 50));
  }

  // If the audio is quiet, remove the oldest node to "fade" the effect
  if (octaveAverages[i] <= 150) {
    nodes[i].shift();
  }
}
```

### Understanding Array Manipulation: Push vs. Shift
To manage these objects, we use two fundamental JavaScript array methods:

*   **`push()`**: Adds a new element to the **end** of an array. In our system, this represents the "birth" of a particle when a musical peak is detected.
*   **`shift()`**: Removes the **first** element from an array and moves all subsequent elements one position to the left. In our system, this represents the "death" of a particle, effectively removing the oldest object in the collection.

## Rendering Nested Data Structures [20:00]

Once we have managed the lifecycle of our nodes, we need to render them. Because our `nodes` variable is a nested array (an array containing multiple sub-arrays), we cannot use a single `for...of` loop. We must use **nested loops** to reach the individual objects.

### The Nested Loop Pattern
The first loop iterates through each octave band (the outer array), and the second loop iterates through the individual `Node` instances within that specific band (the inner array).

```javascript
// draw the octave band nodes
push(); // Isolate stylistic changes (stroke, fill) to these objects
for (let arr of nodes) { 
  // 'arr' represents the sub-array for one specific octave band
  for (let el of arr) { 
    // 'el' is the individual Node object
    el.move();
    el.drawShapes(arr); // Pass the array if the node needs to know about its neighbors
    el.drawLineLines(arr);
  }
}
pop();
```

### Debugging Common Errors
When implementing complex nested structures, two common errors often arise:

1.  **`ReferenceError: [Variable] is not defined`**: This often occurs due to simple typos. For example, if you define `nodes` (plural) but attempt to access `node[i]` (singular), the program will crash.
2.  **Scope and Variable Naming**: When using nested loops, it is vital to use distinct variable names (e.g., `arr` for the sub-array and `el` for the element) to avoid confusion and ensure you are calling methods on the correct object.

### Visualizing Data via the Console
When logic isn't appearing as expected on the canvas, use `console.log()` to inspect your data structures. Checking the values in the console allows you to verify if `octaveAverages` contains the expected numerical data before you attempt to use those values to drive complex animations.

## Debugging the Node Lifecycle [22:30]

When moving from simple shapes to a system of objects, the most common error is attempting to push raw data (like a color) into an array that expects class instances.

If you attempt to push only a color value, the system will fail to render because the drawing loop expects an object with methods like `.move()` or `.drawShapes()`.

### Correcting the Node Injection
To fix this, you must instantiate a new instance of your class within the `push()` method.

**Incorrect Approach:**
```javascript
// This only pushes a color, not an object with behavior
if (octaveAverages[i] > 150 && nodes[i].length < 20) {
  nodes[i].push(color((i / octaveAverages.length) * 255, 0, 50));
}
```

**Correct Approach:**
```javascript
// This pushes a new instance of the Node class
if (octaveAverages[i] > 150 && nodes[i].length < 20) {
  nodes[i].push(new Node(color((i / octaveAverages.length) * 255, 0, 50)));
}
```

## Implementing Nested Data Structures [23:30]

To map specific audio frequencies to specific visual elements, we use a **two-dimensional array** (an array of arrays). Each index in the outer `nodes` array corresponds to a specific octave band.

### Managing Octave Bands
The logic follows three steps for every frequency band:
1. **Initialization**: Ensure the sub-array exists for that index.
2. **Creation (The "Burst")**: If the audio amplitude exceeds a threshold, add a new `Node`.
3. **Destruction (The "Fade")**: If the audio amplitude drops below a threshold, remove an old `Node`.

```javascript
// add/remove nodes on octave bands
for (let i in octaveAverages) {
  // 1. Initialize the sub-array if it doesn't exist
  if (nodes[i] === undefined) {
    nodes[i] = [];
  }

  // 2. Add a node if the sound is loud enough and we haven't hit our limit
  if (octaveAverages[i] > 150 && nodes[i].length < 40) {
    nodes[i].push(new Node(color((i / octaveAverages.length) * 255, 0, 50)));
  }

  // 3. Remove a node if the sound is quiet to prevent "residual" objects
  if (octaveAverages[i] <= 150) {
    nodes[i].shift();
  }
}

// Draw the octave band nodes
push();
for (let arr of nodes) {
  for (let el of arr) {
    el.move();
    el.drawShapes(arr);
  }
}
pop();
```

## Fine-Tuning the Generative Behavior [25:00]

The "feel" of a generative system depends on the relationship between the audio threshold and the array management.

### Solving the "Empty Array" Problem
A common logic error occurs when the threshold for adding a node is higher than the threshold for removing one, but the initial state of the array prevents it from ever growing. 

If you have a condition like `if (nodes[i].length < 20)`, but the array starts at length `0`, and your "add" threshold is too high, you might never trigger the creation of the first node.

### Using `shift()` for Rapid Cleanup
The `.shift()` method removes the first element of an array. In a generative system, this acts as a "cleanup" mechanism to ensure that once the music stops or quietens, the visual elements disappear.

If you find that "residual" objects (visual artifacts that stay on screen too long) are cluttering the canvas, you can call `.shift()` multiple times in a single frame to clear out clusters of objects more aggressively:

```javascript
if (octaveAverages[i] <= 150) {
  nodes[i].shift();
  nodes[i].shift();
  nodes[i].shift();
}
```

### Testing with `sound.jump()`
When debugging complex generative systems, it can be difficult to wait for the right musical moment to see if your code works. Use `sound.jump(seconds)` to skip to a high-energy part of your audio file. This allows you to immediately observe if nodes are being created and, more importantly, if they are being removed correctly when the amplitude drops.

**Visual Result:**
When implemented correctly, you will see a "throbbing" or "spraying" effect where colorful shapes emerge from the vertical position of their corresponding frequency band and are cleared away as the sound fades.

# Transitioning from Spectrograms to Object-Oriented Particle Systems

This lesson explores how to evolve a standard audio spectrogram—which simply maps frequency amplitude to visual height or size—into a generative particle system using Object-Oriented Programming (OOP). By moving from direct data mapping to managing discrete objects, you can create more complex, emergent visual behaviors.

## Refining the Node Class [29:14]

To move away from simple shapes that "explode" randomly across the screen, we need to give our `Node` class more control over its initial state. Instead of using random coordinates, we want the particles to emit from a specific central axis or location related to the audio frequency.

To achieve this, we update the `Node` constructor to accept specific coordinates and an amplitude value. The amplitude is used to scale the velocity, ensuring that louder sounds result in faster-moving particles.

```javascript
class Node {
  constructor(c, x, y, amp) {
    this.x = x;
    this.y = y;
    // We multiply the random velocity by a factor of the amplitude 
    // to create variation in speed based on sound intensity.
    this.xSpeed = random(-1.5, 1.5) * amp / 100;
    this.ySpeed = random(-1.5, 1.5) * amp / 100;
    this.color = c;
  }

  move() {
    // Basic boundary collision logic to keep particles on screen
    if (this.x < 0 || this.x > width) {
      this.xSpeed *= -1;
    }
    if (this.y < 0 || this.y > height) {
      this.ySpeed *= -1;
    }
    this.y += this.ySpeed;
  }
}
```

## Mapping Audio Data to Object Lifecycles [28:48]

Once the `Node` class is capable of receiving position and amplitude, we can use the audio data to trigger the creation and destruction of these objects. This creates a "burst" effect where particles appear when the music hits a certain threshold and disappear as the sound fades.

In `sketch.js`, we iterate through our octave bands. We use a nested array structure (`nodes[i]`) where each index corresponds to an octave band.

```javascript
// add/remove nodes on octave bands
for (let i in octaveAverages) {
  // Initialize the sub-array for this octave if it doesn't exist
  if (nodes[i] === undefined) {
    nodes[i] = [];
  }

  // If the amplitude is high enough, add a new particle
  if (octaveAverages[i] > 150 && nodes[i].length < 40) {
    nodes[i].push(
      new Node(
        color((i / octaveAverages.length) * 255, (i / octaveAverages.length) * 55, random(100)),
        width / 2,                                // Emit from center X
        (i / octaveAverages.length) * height,    // Emit from the Y position of the band
        octaveAverages[i]                         // Use amplitude to drive speed
      )
    );
  }

  // If the amplitude drops, remove old particles to prevent memory leaks
  if (octaveAverages[i] <= 150) {
    nodes[i].shift(); // Removes the oldest element from the array
  }
}
```

### Key Implementation Insights:
*   **Coordinate Mapping:** By using `(i / octaveAverages.length) * height`, we map the particle's starting position to the exact vertical location of its corresponding frequency band. This creates a visual connection between the "source" of the sound and the particle's origin.
*   **Array Management:** Using `.shift()` is essential for performance. Without removing old objects, the `nodes` array would grow indefinitely, eventually slowing down the browser.
*   **Color Variation:** To avoid a monochromatic look, we can map the frequency index to different color channels (e.g., varying the blue channel based on `i`) or use `random()` to add subtle color shifts.

## Moving Toward Generative Art [32:30]

A common pitfall in audio visualization is being "too literal." A standard graphic equalizer (drawing rectangles for every bin) can feel predictable and mechanical. 

By replacing the literal spectrogram with a system of `Nodes`, you shift from **reactive** art (sound $\rightarrow$ movement) to **generative** art (sound $\rightarrow$ seed for complex behavior). The audio no longer just moves a bar; it acts as the energy source that births and kills complex geometric patterns.

The final goal is to move away from simple shapes like `circle()` or `rect()` and instead use the class methods (like `drawShapes` or `drawLines`) to create intricate, crystalline, or organic structures that respond dynamically to the rhythm and intensity of the music.
